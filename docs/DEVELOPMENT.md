# Development strategy & branch plan

A phased, dependency-ordered plan. Each phase produces something runnable and testable
before the next builds on it. Build the **skeleton + foundation first**, then auth, then the
core learning loop, then AI, then payments, then hardening.

## Git workflow

- Long-lived branches: **`main`** (releasable) and **`develop`** (integration).
- Every unit of work branches off `develop` and merges back via PR (squash) once CI passes.
- Naming: `feat/<area>-<slug>`, `fix/<slug>`, `chore/<slug>`, `refactor/<slug>`, `docs/<slug>`.
- Migrations are reviewed with their feature PR; never edit an applied migration — add a new one.

```
main
 └── develop
      ├── feat/...
      └── fix/...
```

## Phase 0 — Foundation (do first)

Make the skeleton boot, connect to Postgres/Redis, and run migrations.

- `chore/project-bootstrap` — uv project, pyproject, ruff/mypy/pre-commit, `.env.sample`.
- `chore/docker-compose` — Postgres, Redis, MinIO, Kafka, app, worker, beat.
- `feat/core-config` — `core/config/*` settings aggregator.
- `feat/core-exceptions-logging` — exception hierarchy + handlers, structlog/loguru.
- `feat/db-foundation` — async engine/session, `models/base.py`, base repository.
- `feat/alembic-setup` — alembic env + first autogenerated migration.
- `feat/app-factory-healthcheck` — `main.py`, router wiring, `/health` + `/ready`.

**Exit:** `uv run python -m app.main` serves `/health`; `alembic upgrade head` works.

## Phase 1 — Auth & security

- `feat/user-model` — `users` table + repository + schemas.
- `feat/auth-jwt` — password hashing, access/refresh issue + rotation, `get_authenticated_user`.
- `feat/auth-register-login` — register/login/logout/refresh endpoints.
- `feat/auth-email-verification` — verification tokens + email send.
- `feat/auth-password-reset` — reset request/confirm flow.
- `feat/auth-google-oauth` — Google OAuth login + account linking.
- `feat/auth-sessions-devices` — device/session registry, list/revoke.
- `feat/rbac` — roles + `require_roles` dependency.
- `feat/security-middleware` — rate limiting / IP throttling, security headers, CSRF, audit log.
- `feat/auth-2fa` *(later)* — TOTP enroll/verify. (gated behind a feature flag)

**Exit:** a user can register, verify, log in (password + Google), refresh, and manage sessions.

## Phase 2 — Vocabulary capture

- `feat/vocabulary-models` — `words`, `sources`, `word_contexts`, `user_words` + repositories.
- `feat/capture-endpoint` — `POST /capture` + `CaptureService` (normalize, persist, dedup).
- `feat/vocabulary-api` — list/search user words, word detail, manual add/remove.

**Exit:** the extension's JSON is accepted and produces vocabulary + context rows.

## Phase 3 — Learning core (no AI yet)

- `feat/learning-models` — exercises, attempts, sessions, review schedule.
- `feat/spaced-repetition` — SM-2 scheduler + `review_schedule` queue.
- `feat/exercise-generators` — fill-in-blanks, match pairs, multiple choice, flashcards,
  sentence reconstruction, timed challenge, listening (template/heuristic versions).
- `feat/learning-sessions` — start session, fetch due items, submit attempts, grade + update SRS.
- `feat/progress-tracking` — streaks, mastery counts, retention metrics.
- `feat/realtime-websocket` — push exercise-ready / progress events.

**Exit:** a full study loop works end-to-end without AI.

## Phase 4 — AI features

- `feat/ai-client` — Anthropic/OpenAI wrapper (retries, timeouts, structured output, token log).
- `feat/ai-context-analysis` — resolve word sense in captured context.
- `feat/ai-difficulty-estimation` — per-user difficulty scoring + adaptive difficulty.
- `feat/ai-exercise-generation` — AI contextual exercises + dynamic quizzes.
- `feat/ai-explanations` — explanations/hints for words and mistakes.
- `feat/ai-learning-path` — personalized path generation from weaknesses + goals.
- `feat/async-ai-jobs` — move AI work to Celery + emit events; persist `ai_generations`.

**Exit:** capture triggers AI analysis; AI-generated exercises and quizzes are served.

## Phase 5 — Payments & subscriptions

- `feat/payment-models` — plans, subscriptions, payments, invoices, promo codes,
  webhook events, usage limits.
- `feat/payment-provider-abstraction` — `PaymentProvider` protocol + base.
- `feat/stripe-provider` — first concrete provider + checkout.
- `feat/subscription-state-machine` — transitions + lifecycle (trial, active, past_due, cancel).
- `feat/webhooks-idempotency` — `/webhooks/{provider}` verify + dedup + dispatch events.
- `feat/billing-invoices` — invoice generation + PDF + email.
- `feat/promo-codes` — validation, redemption limits.
- `feat/usage-limits` — enforce plan quotas (free vs premium).
- `feat/failed-payment-retries` — Celery retries, dunning, auto-renewal.
- `feat/paypal-provider`, `feat/googlepay-provider`, `feat/blik-provider` — additional providers.

**Exit:** a user can subscribe (Stripe), webhooks update state idempotently, limits enforced.

## Phase 6 — Hardening & production

- `feat/observability` — OpenTelemetry traces, Prometheus metrics, Sentry.
- `feat/event-bus-kafka` — promote in-process events to aiokafka topics + consumer service.
- `feat/feature-flags` — runtime toggles.
- `chore/k8s-manifests` — deployments, HPA, ingress, migration job.
- `chore/ci-cd` — lint/type/test + image build + deploy pipeline.
- `test/coverage` — unit (SRS, state machine, generators), integration, e2e.
- `feat/object-storage` — S3/MinIO for TTS audio + exports.

**Exit:** observable, autoscaling, CI/CD-deployed service.

---

## Recommended order (TL;DR)

`Phase 0 → 1 → 2 → 3 → 4 → 5 → 6`. Phases 4 and 5 are independent after Phase 3 and can be
parallelized across people. Within each phase, branches are listed roughly in dependency order.
